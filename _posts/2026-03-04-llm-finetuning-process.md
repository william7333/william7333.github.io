## 왜 파인튜닝(Fine-Tuning)이 필요한가?

프롬프트 엔지니어링 만으로는 특정 도메인(예: 법률, 의료, 자사 특화 데이터)의 전문 용어 지식과 출력 포맷을 100% 강제하기 어렵습니다. 또한 In-Context Learning으로 많은 컨텍스트를 주입하게 되면 토큰 비용(Cost)과 지연율(Latency)이 기하급수적으로 증가합니다.

우리 팀은 8B~14B 파라미터 크기의 소형 LLM(SLM)을 파인튜닝하여, 70B 이상의 상용 모델(GPT-4 등)과 맞먹는 특정 도메인 정확도를 달성했습니다.

## 핵심은 모델이 아니라 '데이터 품질'

"Garbage In, Garbage Out". 너무 진부한 이야기지만, 파인튜닝에서는 이것이 진리입니다.

### 1단계: 데이터 큐레이션 및 정제
* **중복 제거 (Deduplication)**: MinHash 류의 알고리즘을 사용해 의미적으로 동일한 질문-답변(QA) 쌍을 걷어냅니다. 편향된 학습을 막아줍니다.
* **노이즈 필터링**: 특수 문자, HTML 태그, 불완전한 문장 끝맺음을 철저히 파싱하여 제거합니다.

### 2단계: 최적의 QA 페어 생성 파이프라인
초기에는 사람이 직접 레이블링을 하지만, 확장을 위해 상위 모델(Teacher Model)을 활용하여 합성 데이터(Synthetic Data)를 생성합니다. 이때 **Self-Instruct** 기법과 품질 검증 필터(Reward Model 기반)를 통과한 데이터만 학습셋으로 선정합니다.

## 파라미터 효율적 파인튜닝: LoRA 와 QLoRA

자원이 한정된 환경에서는 전체 가중치를 업데이트하는 Full Fine-Tuning 대신, **LoRA (Low-Rank Adaptation)** 를 사용했습니다. 
기존 모델의 가중치는 동결한 채, 저차원 행렬만 추가로 학습시키는 방식입니다. 
더 나아가 **QLoRA**를 적용하여 4-bit 양자화된 베이스 모델 위에서 LoRA 어댑터를 학습시킴으로써 단일 24GB VRAM GPU 환경에서도 7B 모델 튜닝이 가능했습니다.

## 벤치마크 결과 및 회고
파인튜닝 후, 사내 비즈니스 로직 추출 태스크에서 정확도가 **42% -> 91%** 로 상승했습니다. 다음 포스트에서는 이렇게 학습된 모델을 실제 프로덕션 서버에 어떻게 올리는지(vLLM 도입기)를 다루겠습니다.
